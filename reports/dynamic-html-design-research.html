Dynamic HTML / Interactive Artifact Research

Kimi Web 里嵌入可交互 HTML 组件或 iframe 页面的功能设计方案

目标不是把任意 HTML 直接塞进聊天内容，而是把“模型生成的可交互页面”变成一个有边界的 artifact： 可预览、可放大、可下载、可迭代，同时不能拿到宿主页面、daemon API、cookie 或本地文件权限。

调研日期：2026-06-11 仓库：/Users/moonshot/code/kimi-code-web 范围：apps/kimi-web + daemon + protocol

结论

推荐先做 artifact iframe，也就是“消息里出现一个 HTML artifact 卡片，卡片内用受限 iframe 预览，完整内容由 daemon 以独立 artifact 资源提供”。 不建议第一版做 WebContainer / VM 级 app preview，也不建议把 HTML 交给 Markdown 渲染器或 `dangerouslySetInnerHTML`。

1

先支持单文件 HTML

允许 CSS、内联 JS 和小型数据，先不支持安装依赖、跑 dev server、访问本地文件。

2

隔离靠 iframe sandbox

iframe 只开 `allow-scripts`，不加 `allow-same-origin`，网络默认关闭，桥接能力按白名单开放。

3

协议上新增 artifact

当前 message content 没有 artifact 类型；为了保留顺序、版本和持久化，应成为正式内容类型。

我们的现状

当前前端渲染链路

`apps/kimi-web` 现在是 Vue UI components -> `useKimiWebClient` -> daemon adapter -> daemon。 `ChatPane.vue` 负责按 turn 渲染，`Markdown.vue` 用 `markstream-vue` 负责文本和代码块。

动态 HTML 不应进入 `Markdown.vue`。它需要单独的消息块、单独的 iframe 组件和单独的安全策略。

当前协议缺口

`packages/protocol/src/message.ts` 的 `MessageContent` 只有 `text`、`tool_use`、`tool_result`、`image`、`file`、`thinking`。 这意味着 artifact 如果要可持久化、可回放、可按顺序渲染，需要新增正式类型。

用 `metadata` 临时挂 artifact 会让排序、复制、回放和 streaming 处理变复杂；更适合放在 `content` union 里。

实现选型

A. 直接把 HTML 插到宿主 DOM

优点实现最短，静态 HTML 片段显示最快。
问题脚本、样式、事件、表单和宿主页面混在一起；一旦支持 JS，风险不可控。
结论不推荐 只适合纯静态、已严格 sanitize 的小片段，不适合 dynamic HTML。

B. 前端直接用 iframe `srcdoc`

优点最小可验证，适合 spike；iframe 可以 sandbox；不用新增 daemon route。
问题持久化、版本、资源路径、CSP header、下载和回放都比较弱；postMessage 的 origin 是 opaque，需要额外 token 校验。
结论适合技术验证 可以作为 Phase 0，不建议作为产品最终形态。

C. daemon 提供 artifact 资源，前端 iframe 渲染

优点有正式 ID、版本、权限、CSP header、下载和审计；前端只需要渲染 artifact block。
问题需要改 protocol、daemon route/store、web mapper 和 ChatPane。
结论推荐 MVP 成本可控，而且后续能平滑加桥接、版本和资产包。

D. 每个 session 一个 VM / WebContainer 运行环境

优点能跑完整项目、依赖、API route、dev server，接近 v0 的 preview。
问题成本、权限、生命周期、网络、日志、资源限制都明显更重；浏览器内方案还有 COOP/COEP 约束。
结论后续能力 等真实需求变成“跑项目”而不是“展示交互 HTML”时再做。

模块 要做什么 为什么放这里
packages/protocol 新增 `artifact` content schema，REST/WS 都返回同一结构。 artifact 是对话内容的一部分，需要能持久化、回放、排序。
packages/services 新增 `IArtifactStore`，或第一版复用 file blob + artifact index。 保存 HTML 源码、版本、session 归属、size、hash、created_at。
packages/daemon 新增 `/artifacts/{artifact_id}/versions/{version}` 路由，返回 HTML + CSP header。 iframe 不直接读 file route；artifact route 可以加安全 header、权限和下载模式。
apps/kimi-web/src/api/daemon/mappers.ts 把 wire `artifact` 转成 app-facing camelCase 类型。 保持 Vue 层不碰 wire snake_case。
messagesToTurns.ts 把 artifact 保留为 ordered block，跟 text/tool/thinking 保持原始顺序。 一轮里可能先解释、再生成 artifact、再总结，不能 hoist。
ChatPane.vue 遇到 artifact block 渲染 `ArtifactBlock`。 聊天层只负责布局；iframe 细节放独立组件。

安全边界

iframe 默认策略

<iframe
  title="HTML artifact preview"
  sandbox="allow-scripts"
  referrerpolicy="no-referrer"
  loading="lazy"
  src="/artifacts/art_123/versions/1"
></iframe>

不加 `allow-same-origin`。同源 iframe 同时加 `allow-scripts` 和 `allow-same-origin` 会让 sandbox 基本失去意义。

daemon 返回的 CSP 建议

Content-Security-Policy:
  default-src 'none';
  script-src 'unsafe-inline';
  style-src 'unsafe-inline';
  img-src data: blob:;
  font-src data:;
  connect-src 'none';
  frame-src 'none';
  base-uri 'none';
  form-action 'none'

`script-src 'unsafe-inline'` 允许 artifact 内部交互，但风险被限制在 sandbox iframe 里。不要同时开放 daemon API、cookies、同源 DOM 或任意网络。

风险 后果 处理方式
HTML 访问宿主页面 偷读聊天内容、操作宿主 UI、调用 daemon。 iframe sandbox，不加 `allow-same-origin`；宿主不暴露全局 API。
HTML 联网泄漏数据 artifact 可以把对话或用户输入发到外部域名。 第一版 `connect-src 'none'`，图片也只允许 `data:` / `blob:`；需要外链时走显式 allowlist。
postMessage 被伪造 恶意 frame 或页面向宿主发起动作。 校验 `event.source`、消息 schema、artifact instance token；不要信任 `data` 里的任意字段。
资源过大或死循环 页面卡顿、内存膨胀、移动端不可用。 限制 HTML 大小、iframe 数量、预览高度；错误和超时显示 fallback；默认 lazy load。
伪造登录/支付 UI 用户误以为是 Kimi Web 原生界面。 artifact 外壳始终显示标题、来源、sandbox 标识；禁止无提示全屏，外链由宿主确认后打开。

落地路线

Phase 0

前端 spike

用固定 fixture HTML 做 `ArtifactIframe.vue`，验证 sandbox、resize、error、copy source、mobile 布局。不改协议，不接模型。

Phase 1

正式 artifact 类型

新增 protocol content、daemon artifact route/store、web mapper、`TurnBlock` 和 `ArtifactBlock`。只支持完整 HTML 一次性生成，不做 streaming patch。

Phase 2

最小 bridge

开放 `ready`、`height`、`error`、`requestFix`。`requestFix` 只让宿主生成一条用户可见的 follow-up，不让 iframe 直接调 agent。

Phase 3

版本和资产包

支持 artifact version selector、download、source diff、小型 asset bundle。继续禁止外部 CDN，提供内置 utility CSS 方便模型生成稳定 UI。

Phase 4

完整 app preview

只有当需求变成“运行项目 / 安装依赖 / dev server / API route”时，再评估 v0-style sandbox、远端 VM 或 WebContainer。
