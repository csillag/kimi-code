kimi-web「Dynamic HTML」功能调研

在聊天页面中嵌入可交互 HTML 组件 / iframe 页面 · 同类产品方案对比与落地建议 · 2026-06-11

结论先行：推荐 「sandboxed iframe + srcdoc」内联 artifact 卡片 起步（Claude Artifacts / Open WebUI 同路线）， 分三期演进：① files 标签页与工具卡片的 HTML 预览 → ② 聊天流内的 artifact 卡片 + 右侧分屏 → ③ 真实 dev server 嵌入与 postMessage 交互桥。 不需要 WebContainers / Sandpack 这类「浏览器里跑 Node」的重型方案——我们有本地 daemon，能跑真实进程，这是和云端 SaaS 产品最大的架构差异。 落地前必须先解决一个安全前提：daemon API 目前无鉴权、无 CORS 校验，绝不能让模型生成的 HTML 与应用同源运行。

1. 我们要解决的问题

kimi-web 的模型经常产出两类「应该被看到而不是被读源码」的内容：

  • 自包含 HTML 文件：调研报告、方案对比、UI 原型、临时小工具（仓库 docs/ 下已有十几个这样的文件，模型的 HTML 生成规范也明确鼓励这种产物）。目前用户只能在 files 标签页看到源码，或手动去浏览器打开文件。
  • 消息里的 ```html 代码块：目前只做语法高亮（markstream-vue + shiki），无法运行。

目标：这些内容能直接在页面里以可交互的形式渲染，类似 Claude Artifacts 的体验。

产品渲染方式隔离手段支持内容对我们的参考价值

规律很明显：除了要跑完整项目的产品，所有人都收敛到了「sandboxed iframe」这一个原语上；差异只在 HTML 从哪来（srcdoc 内联 / 独立 origin 伺服）和 CSP 收多紧。

3. 我们的架构现状（决定方案的约束）

  • daemon 已能按 Content-Type 伺服任意工作区文件：GET /api/v1/sessions/{id}/fs/*:download（fs.ts:285-308），返回原始字节 + ETag。理论上 iframe src 可以直接指它——但见下面的安全问题。
  • daemon API 无鉴权、无 CORS 配置（auth.ts 只有 readiness probe；未集成 @fastify/cors）。
  • 生产形态下 web UI 与 API 同源（daemon 的 webAssets 路由伺服前端）。
  • 前端已有 FilePreview（markdown/json/图片/代码，无 HTML）、ToolCall 卡片（可从 write/edit 参数提取文件路径）、files 标签页双栏布局，docs/ 里已有 split-layout 的设计探索。
关键安全约束（方案否决项）：模型生成的 HTML 是不可信内容（提示注入可控制它）。如果让它与应用同源运行 （例如 iframe src 直接指向 daemon 的 :download 路由，或 srcdoc 加 allow-same-origin）， 它就能用 fetch 调用无鉴权的 daemon API——读写任意文件、创建会话执行命令，等价于远程代码执行。 所以：渲染不可信 HTML 必须用不带 allow-same-origin 的 sandbox（opaque origin），且永远不要给「同源放行」开关 （Open WebUI 提供了这个开关，是它的减分项）。
顺带发现的存量风险（与本功能无关但建议排期）：daemon 监听 127.0.0.1 且无鉴权，任何本机网页都可以向它发请求（drive-by 攻击面）。 Chrome 的 Private Network Access 只挡「公网页面→本地」的部分场景。建议后续给 daemon 加启动 token（URL 带 token，前端持有，API 校验 header）。

4. 候选方案

方案 A：sandboxed iframe + srcdoc（内联 artifact）推荐起步

前端已经通过 fs:read 拿到 HTML 文本 → 直接塞进 <iframe srcdoc sandbox="allow-scripts allow-forms allow-modals allow-popups">。 不带 allow-same-origin → iframe 是 opaque origin，访问不了 daemon API、localStorage、cookie。

实现量最小：FilePreview 加一种类型 + ToolCall 卡片加一个「预览」按钮即可

安全模型最清晰，和 Claude/Open WebUI 同路线

覆盖我们 90% 的真实场景（自包含 HTML 报告/原型）

页面里的相对路径资源（同目录图片等）加载不了——自包含 HTML 不受影响

opaque origin 下部分 API 受限（如 localStorage 抛错），需在生成规范里说明

方案 B：daemon 独立预览 origin（伺服工作区文件）

daemon 增加一个「预览专用」HTTP 端口或路径（如 127.0.0.1:7879），带严格 CSP 响应头伺服工作区静态文件；iframe src 指向它。 与应用端口不同 → 跨 origin 隔离，相对路径资源也能加载。LibreChat 的 static-browser-server 是这个思路的成熟版（每个预览一个独立 origin）。

支持多文件页面（HTML + 同目录 CSS/JS/图片）

天然支持「在新标签页打开」分享给浏览器

需要 daemon 起第二个 listener + 路径安全（防目录穿越）+ 仍要防它反过来打主 API（主 API 加 origin 校验/token 才算闭环）

实现量中等，但收益只在多文件场景

方案 C：嵌入真实 dev server（iframe src = localhost:port）

模型/用户在工作区跑 vite dev 等，daemon 已知后台任务的端口 → 聊天侧栏出现「预览」卡片，iframe 直接嵌 http://localhost:5173。 这是 bolt.new 体验在本地架构下的「免费」等价物——别人要 WASM 模拟 Node，我们直接用真进程。

支持任意框架的完整应用（HMR 都是真的）

和已有的 web-terminal / split-view 探索方向天然契合

dev server 是用户自己的代码，同 localhost 不同 port 是跨 origin，风险可接受；但 X-Frame-Options/frame-ancestors 可能拒绝被嵌，需降级为「新标签页打开」

依赖端口发现（解析后台任务输出或约定端口）

方案 D：Sandpack / WebContainers不推荐

浏览器内打包运行 React 多文件项目（LibreChat / bolt.new 路线）。

引入庞大依赖（bundler 自托管、Service Worker、独立 origin 服务）

解决的核心问题（无本地执行环境）我们不存在——daemon 能跑真 dev server（方案 C 覆盖同场景且更真实）

5. 推荐的分期落地

Phase 1 — HTML 预览（方案 A，~1-2 天）

模型 write 了 .html / 消息含 ```html→ ToolCall 卡片出现「预览」按钮→ files 标签页 FilePreview 渲染 sandboxed iframe→ 源码/预览切换 + 新窗口打开（Blob URL）
  • FilePreview 增加 html 类型：「预览 | 源码」两个 tab，预览即 srcdoc iframe，高度自适应/可拖拽。
  • ToolCall（write/edit，路径以 .html 结尾）卡片加「预览」入口，跳到 files 标签页选中该文件（toolMeta 已能提取路径）。
  • 「在新窗口打开」用 URL.createObjectURL(new Blob([html]))——blob origin 同样隔离且不经过 daemon。

Phase 2 — 聊天流内 artifact 卡片 + 分屏（~3-5 天）

  • ```html 代码块超过阈值（如 30 行）时折叠为 artifact 卡片：标题 + 预览缩略 + 「展开/分屏打开」。markstream-vue 需要自定义代码块渲染器（调研其扩展点，不行就在 TurnBlock 层先拆出来）。
  • 右侧分屏复用 docs/ 里 split-layout 的设计探索；与 files/diff 面板共享布局。
  • 流式期间只显示「生成中」骨架，结束后再渲染 iframe（避免每个 delta 重建 iframe）。

Phase 3 — 交互桥 + 真实应用预览（按需）

  • postMessage 桥：参考 MCP Apps 的 JSON-RPC over postMessage——artifact 里的按钮可以把消息发回聊天（「基于这个原型改 X」），这是 artifact 从「看」到「用」的关键一步，也为将来直接支持 MCP Apps 留接口。
  • 方案 C：daemon 后台任务暴露端口元数据 → 聊天里出 dev server 预览卡。
  • 安全闭环：daemon 加启动 token + Origin 校验（独立于本功能，建议先做）。
